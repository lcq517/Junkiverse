Stage 00 PHASE 0

环境准备与项目初始化

创建 GitHub 仓库、在 TRAE Work 中初始化项目、配置云端运行环境、安装必备技能。

1

创建 GitHub 仓库

在 GitHub 上创建一个新仓库，用于存放全部前后端代码。建议命名：

Junkiverse

初始化时勾选 "Add a README file"，选择 .gitignore 模板为 Java。

💡 提示：建议仓库结构设为 monorepo，即客户端和服务端代码放在同一仓库的不同目录下，方便 AI 同时理解前后端上下文。
2

安装并打开 TRAE Work

前往 TRAE Work 下载页 下载桌面版，或直接使用 网页版。登录账号后，在界面左上角切换至 Code 模式。

💡 建议桌面版：桌面版支持本地和云端双环境，适合长期复杂项目开发。网页版适合外出时快速查看任务进度。
3

集成 GitHub 到 TRAE Work

在界面左下角点击 头像 → 设置 → 外部应用授权，找到 GitHub 应用并点击 连接。按页面提示登录 GitHub 账号并授权 TRAE-AI CN 访问你的代码仓库。

✅ 完成后：你可以在 TRAE Work 中直接选择 GitHub 仓库作为项目开发，后续也可以让 AI 自动创建 PR。
4

在 TRAE Work 中打开项目

桌面版：打开从 GitHub 拉取的项目。在对话框左下方选择运行环境为云端，然后选择你创建的仓库。

网页版：直接在对话框中发送指令，网页版会自动为你创建项目。

5

配置云端运行环境

本项目需要 Java 运行环境（Spring Boot 后端 + Android 客户端编译）。在 头像 → 设置 → 云端运行环境 中点击 创建，配置如下：

环境名称：Waste Alien Dev 预装依赖：Java 17 环境变量： SPRING_PROFILES_ACTIVE=dev SERVER_PORT=8080 运行时配置： install: mvn clean install -DskipTests start: mvn spring-boot:run
💡 说明：Java 17 是 Spring Boot 3.x 的最低要求，也是 Android 编译的兼容版本。云端环境会自动安装 Maven 依赖并启动后端服务。
6

安装必备技能

在左侧任务栏点击 技能，进入技能市场，搜索并安装以下技能：

技能名称用途
brainstorming需求细化与设计方案梳理，阶段1使用
git-commit生成规范的 Git 提交信息，全流程使用
webapp-testing后端 API 自动化测试，阶段5使用
agent-browser后端接口的浏览器端测试，阶段5使用
💡 技能按需加载：技能不会占用上下文窗口，AI 会在需要时自动调用。你也可以在对话框中输入 / 手动调用。
7

设置项目规则

在 头像 → 设置 → 规则 中创建以下全局规则，让 AI 的输出始终符合项目要求：

规则1：所有代码注释和文档使用中文。 规则2：后端代码遵循 Spring Boot 3.x 规范，使用 Java 17 语法。 规则3：Android 客户端使用 Java 原生开发，不使用 Kotlin。 规则4：API 接口返回统一格式的 JSON 响应体 { code, message, data }。 规则5：所有 API 接口需要添加中文注释，说明接口用途和参数含义。
Stage 01 PHASE 1

需求细化与 UI 设计

使用 brainstorming 技能细化需求，用 Design 模式生成 UI 原型，为后续开发奠定设计基础。

1

使用 brainstorming 技能细化需求

在对话框中输入以下指令，启动头脑风暴技能。AI 会通过一系列提问帮你梳理产品需求、核心功能和设计偏好。

/brainstorming 我要开发一款名为"废物星人改造计划"的 Android App。 核心创意：用 AI 复活废弃物品的创意拼贴 App。 核心闭环：拍照抠图 → 2D 拼贴 → AI 生成角色立绘与故事。 技术栈：Java 原生 Android + Spring Boot 后端。 请帮我梳理完整的产品需求、功能模块清单和 MVP 版本范围。
💡 回答技巧：brainstorming 技能会逐个提问（目标用户、核心功能、设计偏好等），建议认真回答每一个问题，这会直接影响后续 Spec 文档的质量。
2

确认需求文档并保存

brainstorming 技能最终会输出一份结构化的需求设计方案。仔细阅读后，可以直接编辑修改，或用自然语言告诉 AI 你希望调整的部分。确认后，这份文档将作为后续 Spec 工作流的输入。

✅ 完成后：你应该得到一份包含以下内容的需求文档：产品定位、MVP 功能清单、核心用户旅程、页面列表、API 接口列表、AI 模型选型建议。
3

使用 Design 模式生成 UI 原型

切换到 Design 模式（界面左上角），为 App 的核心页面生成高保真设计稿。逐个页面发起设计任务：

设计一个 Android App 的首页。 App 名称：废物星人改造计划 主题：旧物改造 + AI 拼贴 + 环保创意 配色：深绿色主色调 + 琥珀色点缀 + 薄荷绿背景 首页内容： - 顶部：App 名称 + logo - 中间：大按钮"开始创作"（拍照入口） - 下方：最近作品横向滚动列表 - 底部：导航栏（首页 / 我的作品 / 发现）

按照同样的方式，依次生成以下页面的设计稿：

  • 拍照抠图页：相机预览 + 拍照按钮 + 抠图结果展示
  • 拼贴编辑器页：部件库 + 画布 + 工具栏（缩放/旋转/删除）
  • AI 生成结果页：角色立绘大图 + 故事档案文字 + 分享按钮
  • 我的作品集页：作品瀑布流 + 每个作品的角色立绘缩略图
  • 发现/社区页：其他用户的创意作品展示
💡 设计迭代：生成设计稿后，可以在画布中选中元素，点击"添加到对话"，用自然语言描述修改需求让 AI 继续迭代。
4

导出设计稿并准备转入 Code 模式

在 Design 模式的画布中，选中所有设计稿，点击顶部导航栏的 ··· → 导出设计文件 → 在 Code 模式中开发。TRAE Work 会自动打开 Code 模式窗口，将设计稿打包为 .zip 并添加到对话输入框。

✅ 完成后：你将获得完整的 UI 设计稿，并已自动转入 Code 模式，准备开始架构规划。
Stage 02 PHASE 2

系统架构规划（Spec 工作流）

使用 Spec 工作流生成完整的三阶段文档（spec.md / tasks.md / checklist.md），这是整个项目开发的路线图。

为什么用 Spec 工作流？

这是一个从零搭建的完整系统，需求范围大、架构决策多。Spec 工作流会先生成完整的大纲文档，确认后再执行，避免开发到一半发现方向偏差。文档会保存在项目的 .trae/specs/ 目录中，作为长期知识资产。

1

发起 Spec 工作流

在 Code 模式的对话框中输入 /，然后从列表中选择 Spec（或直接输入 /spec 并回车）。然后发送你的需求指令：

/spec 我要从零开发一款名为"废物星人改造计划"的 Android App。 【核心创意】 用 AI 复活废弃物品的创意拼贴 App：拍照抠图 → 2D 拼贴 → AI 生成角色立绘与故事。 【技术栈】 - Android 客户端：Java 原生开发（不使用 Kotlin） - 后端 API：Java + Spring Boot 3.x - AI 抠图：对接抠图模型 API - AI 角色生成：对接文生图模型 API - AI 故事生成：对接大语言模型 API - 数据存储：MySQL + 对象存储 【核心功能模块】 1. 拍照抠图模块：CameraX 相机拍照 → 调用后端抠图 API → 返回透明背景 PNG 2. 拼贴编辑器模块：拖拽/缩放/旋转部件 → 组合拼贴 → 保存拼贴原型 3. AI 生成模块：上传拼贴原型 → 调用 AI 生成角色立绘 + 故事档案 4. 作品管理模块：作品列表/详情/删除/分享 5. 用户系统模块：注册/登录/Token 管理 【参考设计稿】 已通过 Design 模式生成了首页、拍照页、拼贴编辑器页、AI 生成结果页、作品集页的 UI 设计稿（见附件）。 请帮我生成完整的系统 Spec 文档，包括： - 系统架构设计 - 前后端模块拆分 - API 接口定义 - 数据库模型设计 - 开发任务拆解 - 验收标准
2

审阅并确认 Spec 文档

AI 会生成三份文档，存放在 .trae/specs/ 目录下：

文档用途
spec.md系统架构大纲：技术选型、模块划分、数据模型、API 设计
tasks.md开发任务列表：按模块拆解的具体开发任务，带优先级和依赖关系
checklist.md验收清单：每个模块的验收标准，用于确认开发是否达标

文档首次创建时，AI 会暂停执行并等待你的确认。你可以：

  • 直接在文档中编辑修改
  • 用自然语言告诉 AI 你希望修改的部分，如："数据库模型请增加一个素材标签表"、"API 设计请增加分页参数规范"
💡 审阅重点：重点检查 API 接口定义是否覆盖所有功能、数据模型是否合理、任务拆解的粒度是否适中（每个任务建议 1-4 小时可完成）。确认后 AI 会按 tasks.md 的顺序逐步执行。
3

确认项目目录结构

Spec 文档确认后，让 AI 先创建项目的 monorepo 目录结构：

请先创建项目的 monorepo 目录结构： waste-alien-revival-plan/ ├── backend/ # Spring Boot 后端 │ ├── src/main/java/ │ ├── src/main/resources/ │ └── pom.xml ├── android/ # Android 客户端 │ ├── app/src/main/java/ │ ├── app/src/main/res/ │ └── build.gradle ├── docs/ # 项目文档 │ └── api-spec.md # API 接口文档 ├── .trae/specs/ # Spec 文档（自动生成） └── README.md
✅ 完成后：你将获得一份完整的系统架构规划文档，后续所有开发工作都会基于这份文档进行。
Stage 03 PHASE 3

后端开发（Spring Boot）

按照 Spec 文档中的任务列表，逐步开发后端 API 服务。建议使用 Plan 工作流管理每个模块的开发进度。

1

初始化 Spring Boot 项目

在 Code 模式中发送以下指令，让 AI 初始化 Spring Boot 项目骨架：

/plan 请在 backend/ 目录下初始化 Spring Boot 3.x 项目，要求： 1. 使用 Java 17 + Maven 2. 引入依赖：Spring Web、Spring Data JPA、MySQL Connector、Lombok、Spring Security（JWT） 3. 配置 application.yml：数据库连接、文件上传大小限制、JWT 密钥 4. 创建基础包结构： - com.wastealien.config # 配置类 - com.wastealien.controller # 控制器 - com.wastealien.service # 服务层 - com.wastealien.repository # 数据访问层 - com.wastealien.entity # 实体类 - com.wastealien.dto # DTO - com.wastealien.common # 通用工具类（统一响应体、异常处理） 5. 实现统一响应体格式 { code, message, data } 6. 实现全局异常处理
💡 Plan 工作流：Plan 适用于中小型功能开发。AI 会先在 .trae/documents/ 下生成 plan.md 计划文档，确认后再执行。比 Spec 更轻量，适合逐模块开发。
2

开发用户认证模块

继续用 Plan 工作流开发用户认证模块：

/plan 开发用户认证模块，包括： 1. 用户实体类 User（id, username, password, avatar, createdAt） 2. 用户注册接口 POST /api/auth/register 3. 用户登录接口 POST /api/auth/login（返回 JWT Token） 4. JWT 工具类：生成 Token、解析 Token、验证 Token 5. JWT 拦截器：拦截需要登录的接口，解析并验证 Token 6. 密码加密：使用 BCryptPasswordEncoder 7. 单元测试：注册和登录接口的测试用例
3

开发素材管理模块（抠图对接）

开发素材管理模块，包括拍照后的图片上传和 AI 抠图对接：

/plan 开发素材管理模块，包括： 1. 素材实体类 Material（id, userId, originalImageUrl, cutoutImageUrl, status, createdAt） 2. 图片上传接口 POST /api/materials/upload（接收 multipart 文件，存储到对象存储） 3. AI 抠图接口 POST /api/materials/{id}/cutout（调用抠图模型 API，返回透明背景 PNG） 4. 素材列表接口 GET /api/materials（分页查询当前用户的素材） 5. 素材删除接口 DELETE /api/materials/{id} 6. 对象存储工具类：封装文件上传、下载、删除操作 7. 抠图 API 对接：设计通用的 AI 服务调用接口，方便后续切换不同抠图模型 注意：抠图模型 API 的地址和密钥放在配置文件中，通过 @Value 注入。 先实现 Mock 模式（直接返回原图），后续再对接真实 API。
4

开发拼贴作品模块

开发拼贴作品模块，管理用户的拼贴创作：

/plan 开发拼贴作品模块，包括： 1. 作品实体类 Artwork（id, userId, title, collageImageUrl, characterImageUrl, storyContent, status, createdAt） 2. 保存拼贴原型接口 POST /api/artworks（接收拼贴图片 URL + 标题，创建作品记录，状态为 pending） 3. 触发 AI 生成接口 POST /api/artworks/{id}/generate（调用 AI 生成角色立绘 + 故事档案） 4. AI 角色生成对接：调用文生图模型 API，基于拼贴原型图片 + Prompt 生成角色立绘 5. AI 故事生成对接：调用大语言模型 API，基于角色立绘图片描述生成幽默档案故事 6. 作品列表接口 GET /api/artworks（分页查询当前用户的作品） 7. 作品详情接口 GET /api/artworks/{id} 8. 作品删除接口 DELETE /api/artworks/{id} 9. 发现/社区接口 GET /api/artworks/discover（分页查询所有用户公开的作品） 注意：AI 生成接口设计为异步模式（先返回 task_id，客户端轮询生成结果）。 先实现 Mock 模式，后续再对接真实 API。
5

开发 AI 服务对接层

统一管理所有 AI 模型 API 的对接，设计可切换的适配器模式：

/plan 开发 AI 服务对接层，设计适配器模式方便后续切换不同模型服务商： 1. 定义统一接口： - ImageCutoutService：抠图服务接口（输入图片 → 输出透明背景 PNG） - ImageGenerationService：文生图服务接口（输入 Prompt + 参考图 → 输出角色立绘） - TextGenerationService：文本生成服务接口（输入图片描述 → 输出故事档案） 2. 实现 Mock 适配器（开发测试用，返回预设图片和文本） 3. 实现真实适配器（后续对接）： - 抠图：对接 rembg / U2Net 模型 API - 文生图：对接豆包 / Seedream 模型 API - 文本生成：对接豆包大模型 API 4. AI 服务配置管理：在 application.yml 中配置当前启用的适配器、API 地址和密钥 5. 异步任务管理：AI 生成任务的状态管理（pending → processing → success / failed）
6

使用 git-commit 技能提交代码

每完成一个模块的开发，使用 git-commit 技能生成规范的提交信息并提交：

/git-commit 请提交当前所有变更，按 Conventional Commits 规范生成提交信息。
💡 提交频率：建议每完成一个功能模块就提交一次，保持每次提交只包含一个逻辑变更。这样后续如果需要回退或查看历史，粒度更合适。
Stage 04 PHASE 4

Android 客户端开发

按照 Spec 文档中的任务列表，逐步开发 Android 客户端。使用 Java 原生开发，不使用 Kotlin。

1

初始化 Android 项目

/plan 请在 android/ 目录下初始化 Android 项目，要求： 1. 使用 Java 原生开发（不使用 Kotlin） 2. 最低 SDK 版本：minSdk 24（Android 7.0） 3. 目标 SDK 版本：targetSdk 34（Android 14） 4. 引入依赖： - CameraX：相机拍照功能 - OkHttp / Retrofit：网络请求 - Gson / Moshi：JSON 解析 - Glide：图片加载和缓存 - RecyclerView：列表展示 - ViewBinding / DataBinding：视图绑定 - Material Components：Material Design 组件 5. 配置 AndroidManifest.xml：相机权限、网络权限、文件存储权限 6. 创建基础包结构： - com.wastealien.ui # Activity / Fragment - com.wastealien.ui.adapter # RecyclerView Adapter - com.wastealien.ui.widget # 自定义 View - com.wastealien.data # 数据层（Repository / API Client） - com.wastealien.data.api # API 接口定义 - com.wastealien.data.model # 数据模型 - com.wastealien.util # 工具类 - com.wastealien.widget # 自定义控件 7. 实现统一的网络请求客户端（Retrofit + OkHttp + 拦截器） 8. 实现统一的 API 响应体解析
2

开发首页和底部导航

/plan 开发 App 首页和底部导航功能： 1. 首页 Activity（HomeActivity）： - 顶部：App 名称 + Logo - 中间：大按钮"开始创作"（点击跳转到拍照页） - 下方：最近作品横向滚动列表（RecyclerView + HorizontalLayoutManager） - 底部：底部导航栏（首页 / 我的作品 / 发现） 2. 底部导航（BottomNavigationView）： - 三个 Tab：首页、我的作品、发现 - 通过 Fragment + NavigationController 实现页面切换 3. 最近作品卡片布局： - 角色立绘缩略图（正方形） - 作品标题 - 点击跳转到作品详情页 4. 参考设计稿实现 UI（见附件 Design 模式生成的首页设计稿）

开发拍照抠图模块

TRAE Work 使用技巧

Spec vs Plan：什么时候用哪个？

维度Spec 工作流Plan 工作流
适用场景从零搭建新系统/模块、大规模重构中小型功能开发、模块级重构
文档复杂度三阶段文档组（spec.md + tasks.md + checklist.md）单份计划文档（plan.md）
文档存放.trae/specs/ 目录.trae/documents/ 目录
本项目中推荐使用阶段 2（系统架构规划）阶段 3-4（各模块开发）

如何高效地与 AI 对话？

  • 提供足够的上下文：每次发起 Plan 时，附上相关的 Spec 文档、设计稿、数据库模型等上下文信息，让 AI 的规划更准确
  • 明确技术约束：在指令中明确说明技术栈、代码规范、接口格式等约束条件，避免 AI 生成不符合预期的代码
  • 分模块逐步开发：不要一次性让 AI 开发所有功能，而是按模块逐个发起 Plan，每个模块完成后测试再继续下一个
  • 及时反馈问题：如果 AI 生成的代码有问题，直接在对话中指出具体问题和期望的修改方式，AI 会迭代修改
  • 善用 /git-commit 技能：每完成一个功能模块就用 git-commit 技能提交代码，保持提交历史规范清晰

云端运行环境使用建议

  • 敏感信息用敏感变量：AI 模型 API 密钥、数据库密码等敏感信息使用"敏感变量"存储，会被 KMS 加密
  • 普通配置用环境变量：DEBUG 开关、LOG_LEVEL 等普通配置使用"环境变量"存储
  • 网络策略配置：如果项目只需要访问特定的依赖源，可以配置网络策略白名单，限制出站流量提升安全性
  • 多服务并行启动：如果项目需要同时启动后端服务和数据库，可以在 terminals 配置中并行启动多个服务

Design 模式与 Code 模式的协同工作流

  • 先 Design 后 Code：先在 Design 模式中完成所有页面的 UI 设计，确认满意后统一导出到 Code 模式开发
  • 设计稿作为上下文：在 Code 模式中开发每个页面时，附上对应的 Design 模式设计稿作为参考
  • 设计迭代与代码迭代分离：UI 需要调整时回到 Design 模式修改，功能逻辑调整时在 Code 模式修改
